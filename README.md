# AI 顾投小助手

AI 顾投小助手是一个嵌入用户买入 / 卖出操作流程的动态风险冲突识别系统。

它解决的核心问题是：用户的资金用途、用款时间、投资目标和风险承受能力会动态变化，但传统风险测评通常是开户或定期填写的静态结果，无法及时识别用户“当前这一次操作”与真实资金安排之间的不匹配。

本项目不替代风险测评、智能投顾或自动资产配置，也不提供确定性的买入 / 卖出建议。它增加的是一层“交易当下的动态冲突识别能力”：在用户发起买入 / 卖出时，结合已确认画像和产品信息，识别资金安排、风险承受能力、产品风险和当前操作之间的冲突，并在必要时提示用户或追问以更新画像。

## 核心价值

针对用户资金安排及风险承受能力动态变化、传统静态测评无法及时识别操作冲突的问题，设计买入 / 卖出环节的动态风险冲突识别能力。

价值分四层：

- 风险管理价值：保留适当性过程留痕，解释为什么提示、提示了什么、用户如何选择。
- 运营效率价值：减少事后投诉、客服解释和人工复核成本。
- 用户体验价值：用具体字段冲突替代抽象风险等级提示，例如“用款时间”和“产品封闭期”不匹配。
- 远期能力输出：未来可作为模块化能力嵌入理财、基金、券商或财富管理平台。

净价值测算框架：

```text
MVP净价值 = 投诉/纠纷成本节约 + 人工处理成本节约 + 留存价值改善
          - 研发与系统成本 - 模型调用成本 - 合规运营成本 - 交易摩擦损失
```

## 当前产品形态

当前版本是本地可运行 MVP，默认进入“产品与交易”页。

用户操作路径：

1. 用户浏览产品。
2. 用户点击买入或卖出。
3. 系统读取用户画像和产品信息。
4. 后端规则引擎执行冲突判断。
5. 页面进入交易确认页。
6. 如果存在中 / 高风险冲突，右下角展示非阻断风险提示。
7. 用户可以继续确认交易，也可以撤回操作。
8. 高冲突时可打开悬浮 AI 对话框回答追问。
9. 用户回答后，系统抽取画像更新建议，经后端校验后更新画像并重新计算冲突。

重要交互原则：

- 冲突提示不阻断交易确认页。
- 中 / 高风险提示都提供“撤回操作”。
- 没有追问时，提示卡显示“撤回操作 / 继续操作”。
- 有追问时，提示卡显示“撤回操作 / 打开对话回答 / 关闭提示”。
- 左侧“对话”栏目不会切换主页面，只打开悬浮 AI 对话框。
- 悬浮对话框中，“回答追问”和“问 AI 一个问题”是两个独立输入框。

## 核心问题

| 问题 | 说明 |
| --- | --- |
| 画像过期 | 用户资金安排在两次正式测评之间可能已经变化 |
| 识别滞后 | 平台仍按旧画像判断当前操作，无法及时发现错配 |
| 业务影响 | 增加用户亏损、投诉、流失，提高合规与客服处理成本 |

## 产品差异化

传统风险提示、智能投顾和组合管理产品，多停留在开户测评、目标设定或定期复测环节。

本产品聚焦于：

- 用户发起具体买入 / 卖出操作的当下。
- 比对已确认画像中的资金安排与本次操作是否冲突。
- 在信息可能过期时主动核实，而不是要求用户重走一遍完整目标设定流程。
- 用确定性规则输出冲突等级，保证可解释、可测试、可审计。

与 Betterment、Wealthfront、Schwab 等目标设定或智能投顾产品相比，本项目增加的是一层动态冲突识别层。

## 系统架构

整体调用流程分为 8 个阶段：

1. 意图与情绪识别：AI 调用，识别 `intent`、`emotion`、`execution_stage`、`confidence`。
2. 画像字段路由：规则，根据意图映射所需画像字段。
3. 用户画像与产品信息获取：服务调用，读取画像与产品数据。
4. 冲突识别：规则引擎输出冲突类型与冲突等级。
5. 识别后回答：AI 生成用户可读回答；高冲突时生成一个关键澄清问题。
6. 用户回答与画像修改：AI 抽取候选画像字段变化。
7. 画像更新：后端校验通过后写入数据库。
8. 冲突重新计算：基于更新后画像重新计算冲突等级。

核心设计原则：AI 与规则严格分层。

AI 负责：

- 意图、情绪、执行阶段识别
- 面向用户的解释性回答
- 高冲突关键澄清问题生成
- 候选画像字段变化抽取

规则和后端确定性代码负责：

- 冲突等级判断
- 意图到画像字段路由
- 字段合法性校验
- 是否允许画像更新
- 画像写入数据库
- 冲突重新计算
- 日志、指标和审计记录

AI 不得：

- 单独决定冲突等级
- 直接修改用户画像
- 修改冲突规则
- 编造缺失画像或产品指标
- 给出确定性的买入、卖出、加仓、减仓或清仓指令
- 承诺收益或预测确定涨跌

## 用户画像体系

画像分为 7 大模块，首次通过问卷和派生算法生成，后续根据追问结果更新：

- 基础信息：应急资金、收入稳定性、负债偿付率等
- 具体目标：目标类型、金额、期限、优先级、可实现度
- 投资期限：使用日期、可延后程度、中途用款概率
- 风险承受能力：最大可承受损失、风险缓冲能力评分
- 资金情况：已准备金额、资金缺口率、目标完成率
- 投资经验：投资年限、产品类型、决策自主程度
- 陪伴需求：长期陪伴偏好、当下情绪与紧急程度

画像派生计算集中在：

```text
src/domain/profile/deriveProfile.ts
```

字段配置位于：

```text
src/config/profileFields.ts
```

## 冲突识别规则

冲突规则位于：

```text
src/domain/conflicts/
```

当前实现的 P0 规则：

| 冲突类型 | 触发逻辑 |
| --- | --- |
| 损失承受-历史回撤冲突 | 产品历史最大回撤超出用户有效最大损失比例 |
| 期限-流动性冲突 | 用户最早可能用款时间早于产品可赎回 / 到账时间 |
| 目标刚性-产品波动冲突 | 高优先级目标与高波动产品不匹配 |
| 中途用款-产品波动冲突 | 中途用款概率高且产品封闭期 / 到账周期长 |
| 风险缓冲弱-持仓风险冲突 | 风险缓冲能力弱且产品回撤超过阈值 |
| 资金缺口-风险承担冲突 | 资金缺口率高且产品回撤超过阈值 |
| 恐慌卖出-长期目标冲突 | 恐慌卖出与长期目标不一致 |
| 兴奋买入-风险承受冲突 | 兴奋买入的产品风险超出承受范围 |

每类冲突分为：

- low
- medium
- high

多个规则同时命中时，系统保存所有命中结果，整体冲突等级取最高等级。

## 冲突响应策略

| 冲突等级 | 会话内 | 不在会话内 / 交易流程 |
| --- | --- | --- |
| 低冲突 | 直接回答，仅记录 | 仅记录，不打扰 |
| 中冲突 | 回答 + 提示卡，不追问 | 提示卡，带撤回 / 继续 |
| 高冲突 | informational_only 回答 + 提示卡 + 一个关键追问 | 提示卡 + 悬浮对话追问入口 + 撤回 |

约束：

- 每个业务请求最多生成一个关键澄清问题。
- 每个业务请求最多尝试一次画像更新。
- 更新后冲突仍未解除，不在同一请求内重复追问。

## AI 模块

| 模块 | 模型方向 | 输出 |
| --- | --- | --- |
| 意图与情绪识别 | 轻量结构化抽取 | intent / emotion / execution_stage / confidence / evidence |
| 用户问题回答 | 通用大模型 | 面向用户的自然语言回答 |
| 高冲突关键追问 | 大模型 + 固定模板 | 一个关键澄清问题 |
| 画像字段更新抽取 | 轻量结构化抽取 | update_required / new_value / status / confidence / evidence |

Provider 目录：

```text
src/lib/llm/
```

默认 Mock Provider 可离线演示；OpenAI-compatible Provider 可通过环境变量启用。

## MVP 范围

当前覆盖：

- 买入、卖出两类交易操作
- 公募基金 / ETF / 理财等结构化产品演示数据
- 用户画像问卷与派生字段
- 确定性冲突规则
- 非阻断交易提示
- 高冲突追问与画像更新校验
- 完整日志审计
- Mock LLM 与 OpenAI-compatible Provider

暂不覆盖：

- 真实登录和真实账户体系
- 真实订单系统、支付、清算、持仓变更
- 自动执行交易
- 个股、期权、私募等高复杂度产品
- 调仓、定投、资产配置推荐
- 多轮追问
- 同一请求多次画像更新

优先级判定公式：

```text
Priority Score = 0.30 * 风险严重程度
               + 0.25 * 发生频率
               + 0.20 * 数据可获得性
               + 0.15 * 规则确定性
               + 0.10 * 实现可行性
```

前端指标页面已移除，但后端保留指标聚合接口，后续可重新接入运营后台。

安全合规：

- 禁止交易指令率

业务效果：

- 中冲突操作放弃率

- 画像有效更新率
- 冲突误报率
- 风险提示理解度

## 灰度路径

建议灰度路径：

1. 单规则小流量验证：期限-流动性冲突。
2. P0 多规则扩大灰度：恐慌卖出、兴奋买入等。
3. P1 高冲突能力灰度：合规确认后开放画像更新。
4. 稳定放量与主要场景覆盖。

每阶段设置明确 Go / No-Go 准入条件，不按固定时间表强行扩量。

## 边界声明

本项目不做：

- 用户自动交易执行
- 个性化买入 / 卖出建议
- 收益预测或确定性涨跌判断
- 外部银行流水或跨平台账户接入
- 大模型单独决定冲突等级
- 大模型直接修改画像
- 合规确认前的生产级画像更新能力

## 当前取舍

- 交易确认提交是 MVP 演示状态，不是真实下单。
- Prisma SQLite 当前版本不支持原生 Json 字段，项目使用 TEXT 保存序列化 JSON。
- 当前环境下 Prisma `migrate dev` 存在 schema-engine 兼容问题，项目通过 `scripts/prisma-wrapper.ts` 将 `pnpm prisma migrate dev` 包装为执行 migration SQL 并生成 Prisma Client。
- 前端指标页面已删除，后端 metrics API 暂保留。

## 生产化建议

- 接入真实登录、账户、持仓、订单和交易确认系统。
- 将撤回 / 继续与真实订单状态机打通。
- 增加不可篡改审计日志。
- 引入真实产品数据源和申赎日历。
- 增加规则版本、字段版本和 Prompt 版本管理。
- 增加端到端浏览器测试、人工复核后台和模型质量评估集。
